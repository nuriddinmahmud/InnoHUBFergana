INNOHUB BACKEND DATA SCHEMA
Generated from current frontend behavior and admin panel requirements
Date: 2026-03-18

==================================================
1. PURPOSE
==================================================

This file describes the backend database structure that matches the current INNOHUB frontend.

Current frontend features covered by this schema:
- Firebase Google authentication
- User profile storage
- Public courses list
- Course detail + lessons
- Lesson progress tracking
- Admin panel:
  - Dashboard
  - Courses CRUD
  - Topics CRUD
  - Users management
  - Statistics

Important note:
- The frontend currently uses static data and localStorage for several features.
- This schema describes how those same features should be stored in a real backend database.

==================================================
3. USERS TABLE
==================================================

Table: users

Purpose:
- Stores the main user record used across dashboard, navbar, admin panel, and future profile pages.

Fields:
- id: uuid or varchar primary key
- firebase_uid: varchar unique not null
- email: varchar unique not null
- full_name: varchar not null
- avatar_url: text null
- role: enum('student', 'admin', 'super_admin') default 'student'
- status: enum('active', 'blocked', 'pending') default 'active'
- provider: enum('google', 'email') default 'google'
- created_at: timestamp not null
- updated_at: timestamp not null
- last_login_at: timestamp null

Frontend mapping:
- AuthContext stores:
  - id
  - email
  - name
  - avatar / photoURL
  - token
- AdminUsers page expects:
  - avatar
  - name
  - email
  - joined date
  - status

Recommended indexes:
- unique(firebase_uid)
- unique(email)
- index(role)
- index(status)
- index(created_at)


==================================================
4. AUTH ACCOUNTS TABLE
==================================================

Table: auth_accounts

Purpose:
- Separates login provider data from the main user row.
- Useful if later one user can connect Google + email/password.

Fields:
- id: uuid primary key
- user_id: fk -> users.id
- provider: enum('google', 'email')
- provider_user_id: varchar not null
- email: varchar null
- access_token: text null
- refresh_token: text null
- token_expires_at: timestamp null
- created_at: timestamp not null
- updated_at: timestamp not null

Notes:
- For Firebase Google login, provider_user_id should usually be firebase_uid.
- If backend verifies Firebase ID token, access_token may be optional to store.

5. COURSES TABLE
==================================================

Table: courses

Purpose:
- Stores all courses shown on `/courses`, `/dashboard`, `/course/:courseId`, and admin course management.

Fields:
- id: varchar primary key
  Examples:
  - html
  - css
  - javascript
- slug: varchar unique not null
- title: varchar not null
  Current frontend titles:
  - HTML Fundamentals
  - CSS Fundamentals
  - JavaScript Fundamentals
- description: text not null
- icon: varchar null
  Example values currently used in frontend:
  - emoji or short label
- image_url: text null
- gradient_from: varchar null
  Example: from-orange-500
- gradient_to: varchar null
  Example: to-red-500
- level: enum('Boshlang''ich', 'O''rta', 'Yuqori') not null
- duration_label: varchar not null
  Example:
  - 6 soat
  - 7 soat
  - 12 soat
- total_lessons: int not null default 0
- status: enum('draft', 'published') default 'draft'
- sort_order: int default 0
- created_by: fk -> users.id null
- created_at: timestamp not null
- updated_at: timestamp not null

Frontend mapping:
- AdminCourses page expects:
  - image
  - title
  - description
  - level
  - duration
  - status
  - topicCount
- Dashboard/Courses pages expect:
  - title
  - description
  - image/icon
  - level
  - totalLessons
  - duration

Recommended indexes:
- unique(slug)
- index(status)
- index(level)
- index(sort_order)

6. TOPICS TABLE
==================================================

Table: topics

Purpose:
- Stores all lessons/topics inside a course.
- One course has many topics.

Fields:
- id: uuid primary key
- course_id: fk -> courses.id not null
- lesson_number: int not null
  Current frontend uses numeric lesson ids 1..35
- title: varchar not null
- video_id: varchar not null
  Current YouTube placeholder: dQw4w9WgXcQ
- duration_label: varchar not null
  Example:
  - 8:24
  - 10:15
- content_markdown: longtext null
- content_html: longtext null
  Optional pre-rendered cache if needed
- is_preview: boolean default false
- status: enum('draft', 'published') default 'published'
- created_at: timestamp not null
- updated_at: timestamp not null

Constraints:
- unique(course_id, lesson_number)

Frontend mapping:
- CoursePage needs:
  - course_id
  - lesson_number
  - title
  - video_id
  - duration_label
  - optional content/description
- AdminTopics page expects:
  - course
  - order number
  - title
  - YouTube ID
  - duration
  - markdown content

Recommended indexes:
- unique(course_id, lesson_number)
- index(course_id)
- index(status)

Fields:

Frontend mapping:

  - total lesson count
  - continue button target

Recommended indexes:
- unique(user_id, course_id)
- index(user_id)
- index(course_id)
- index(status)

8. LESSON PROGRESS TABLE
==================================================

Table: lesson_progress

Purpose:
- Replaces current localStorage progress:
  key format now in frontend: innohub_progress_{courseId}
  structure now in frontend:
  {
    completedLessons: number[],
    lastLessonId: number
  }

Recommended backend row-per-lesson model:

Fields:
- id: uuid primary key
- user_id: fk -> users.id not null
- course_id: fk -> courses.id not null
- topic_id: fk -> topics.id not null
- is_completed: boolean not null default false
- completed_at: timestamp null
- last_viewed_at: timestamp null
- watch_seconds: int default 0
- created_at: timestamp not null
- updated_at: timestamp not null

Constraints:
- unique(user_id, topic_id)

Why this structure is better:
- Easier analytics
- Easier progress calculation
- Supports resume from last lesson
- Supports partial watch time later

How to derive current frontend values:
- completedLessons:
  SELECT topic.lesson_number WHERE is_completed = true
- lastLessonId:
  latest viewed topic.lesson_number for that user/course
- progressPercent:
  completed topics / total topics * 100

Recommended indexes:
- unique(user_id, topic_id)
- index(user_id, course_id)
- index(course_id)

15. MINIMUM BACKEND API CONTRACT
==================================================

These are the minimum endpoints the current frontend would benefit from after moving off static/local data.

Auth:
- POST /auth/firebase
  Verify Firebase token and create/update user
- GET /auth/me
  Return current user profile
- POST /auth/logout

Courses:
- GET /courses
  Return published courses with total_lessons and basic meta
- GET /courses/:courseId
  Return one course with lessons/topics
- POST /admin/courses
- PUT /admin/courses/:courseId
- DELETE /admin/courses/:courseId

Topics:
- GET /admin/topics?courseId=
- POST /admin/topics
- PUT /admin/topics/:topicId
- DELETE /admin/topics/:topicId

Progress:
- GET /me/courses
  Return enrolled courses + progress summary
- GET /me/courses/:courseId/progress
- POST /me/courses/:courseId/enroll
- POST /me/courses/:courseId/topics/:topicId/complete
- DELETE /me/courses/:courseId/topics/:topicId/complete
- POST /me/courses/:courseId/last-viewed

Users/Admin:
- GET /admin/users
- PATCH /admin/users/:userId/status
- DELETE /admin/users/:userId

Statistics:
- GET /admin/statistics/overview
- GET /admin/statistics/users
- GET /admin/statistics/courses

19. IMPORTANT FRONTEND COMPATIBILITY NOTES
==================================================

1. Course IDs are string-based right now.
   Keep stable IDs such as:
   - html
   - css
   - javascript

2. Current frontend expects lesson ordering by integer sequence.
   Keep lesson_number numeric and unique per course.

3. Current frontend stores Google user as:
   {
     id,
     email,
     name,
     avatar,
     photoURL,
     token
   }
   Backend response should map cleanly to this.

4. Current frontend uses Uzbek for many labels but course titles are in English.
   Keep course title fields in English.

5. Current admin pages already assume:
   - course has status
   - topic belongs to one course
   - user has status
   - statistics can be grouped by date and by course
